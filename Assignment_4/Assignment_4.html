  • Microservices (50 points)

    Microservices: Transaction Management with Fiscal Receipt Generation

    Microservices are a software architecture approach where an application is divided into small, independent services. Each microservice is responsible for a specific functionality and communicates with others through APIs. This approach makes it easier to develop, scale, and deploy services independently. For example, if Kaspi Bank builds Kaspi Travel and Kaspi Shop as separate microservices, a failure in one service will not affect the functionality of the others. This ensures that users can still access other features without interruption.

    Goal:

    Create a microservice for managing transactions related to purchasing products, services, or premium subscriptions. The microservice should also generate a fiscal receipt and send it to the customer via email.

    Tasks for the Main Server:

    1. Create a Cart Page (if necessary):

      Add the necessary pages for order processing and create new database tables for the shopping cart.

    1. Add a Payment Button:

      Implement functionality to process payments. For example, allow users to pay for their cart (if it’s an online store) or subscribe to a premium plan (if a store-like logic isn’t applicable).

    1. Create a Transaction Table:

      After the cart is prepared, create a separate table for transactions. This table should include the transaction ID, cart ID, and transaction status (e.g., "Pending Payment," "Paid," "Declined"). This will help track the transaction’s progress and determine whether the payment was successful.

    1. Send a POST Request from the Main Server to the Microservice:

      After the payment button is clicked and a transaction ID is created, send a POST request from the main server to the microservice. The request should include cart and customer details to create a corresponding record in the microservice database. Example request:

      {
  "cartItems": [
    {
      "id": "1",
      "name": "Product 1",
      "price": 100
    },
    {
      "id": "2",
      "name": "Product 2",
      "price": 200
    }
  ],
  "customer": {
    "id": "123",
    "name": "John Doe",
    "email": "john.doe@example.com"
  }
}

      These details will be used in future tasks.

    1. Handle the Microservice Response:

      Based on the response from the microservice (true or false), display the relevant information to the user. For example, update the user’s profile page with the transaction status.

    Tasks for the Microservice:

    1. Create the Microservice:

      Develop a separate microservice using Go (or any other language). Ensure it runs independently of the main server, for example, on port 8081.

    1. Database Management:

      Ideally, the microservice should have its own database. However, for this small project, you can use the same database but ensure the tables are separate. Make sure there are no foreign key dependencies between the main server tables and the microservice tables.

    1. Payment Form:

      After receiving a POST request from the main server, the microservice should present the user with a payment form to input payment details (e.g., card number, expiration date, etc.). For testing purposes, use mock data and return either a successful result (true) or a failed result (false) depending on the input (e.g., expired card data).

      Example of a payment form:

      {
    "paymentForm": {
        "cardNumber": "1234 5678 9012 3456",
        "expirationDate": "12/24",
        "cvv": "123",
        "name": "John Doe",
        "address": "123 Main Street"
    }
}

      The payment result should be sent back to the main server, which will update the transaction table accordingly:

      • "Pending Payment" → "Paid" (if true)
      • "Pending Payment" → "Declined" (if false)
    1. Generate a Fiscal Receipt:

      After a successful payment, generate a fiscal receipt in PDF format. The receipt should include the following details:

      • Company or project name
      • Transaction ID
      • Order date and time
      • Product or service names
      • Price per unit
      • Quantity
      • Total amount
      • Customer's full name
      • Payment method (with encrypted card details)
    1. Send the Receipt:

      Send the generated PDF receipt to the customer’s email using the microservice.

    1. Finalize the Transaction:

      Update the transaction status to "Completed" after successful payment. Ensure that all changes are correctly reflected in both the database and the user interface.

    Goal:

    Enhance the functionality of your current project by implementing a real-time chat feature where clients can interact with support (administrator). This chat can be useful for quick communication, such as resolving issues with orders or subscriptions.

    Steps:

    1. Building on Your Existing Project:

      Use your current project, which already includes an admin panel and a client page, and extend it to include chat functionality.

    1. Core Chat Features:

      To implement the chat, follow these steps:

      1. Assigning a Unique Identifier:

        When a client opens a chat, a unique chat ID should be automatically generated for them.

      1. Limiting to One Active Chat:

        Each client can only have one active chat at a time.

      1. Displaying Chat Links:
        • Clients should see a link to their active chat in their profile.
        • Administrators should see links to chats from different clients in their admin panel.
      1. Saving Chat History:

        Chat messages should be saved so that the history remains accessible even after re-login, page refresh, or session expiration. The chat history should be stored in a database or a text file and tied to the unique chat ID.

      1. Closing Chats by the Administrator:
        • Administrators should have the ability to close a chat with a client.
        • Once a chat is closed:
          • The chat history becomes unavailable.
          • The chat status changes to "Active" or "Inactive."
          • Links to the closed chat disappear from both the admin panel and the client profile.
          • The client sees a button to start a new chat.
      1. Creating a New Chat:
        • After a chat is closed, the client can create a new one.
        • A new chat ID is generated, a corresponding record is created in the database, and the process repeats.
    1. Project Updates:

      Implement the following changes in your project:

      1. Chat Page:
        • The chat should have standard features: showing the sender, timestamp, message content, and a send button.
        • The interface should be simple and user-friendly.
      1. Client Profile Updates:
        • If a client has an active chat, display a link to it.
        • If no chat is active, show a button to create a new chat.
        • Do not display links to inactive chats.
      1. Admin Panel Updates:
        • Add a section or table listing links to all active client chats.
        • Include a button to close each chat.

    Additional Notes:

    • Use the github.com/gorilla/websocket library to work with WebSockets.
    • On the server: create a /ws handler to manage WebSocket connections, sending, and receiving messages.

      Example Code:

      var upgrader = websocket.Upgrader{
    CheckOrigin: func(r *http.Request) bool { return true },
}

func handleConnections(w http.ResponseWriter, r *http.Request) {
    ws, err := upgrader.Upgrade(w, r, nil)
    if err != nil {
        log.Println(err)
        return
    }
    defer ws.Close()
}

      Ensure the server listens on a specific port:

      http.HandleFunc("/ws", handleConnections)
log.Fatal(http.ListenAndServe(":8080", nil))
    • On the client: use the WebSocket API to connect to the server and send/receive messages. In your HTML file, establish the WebSocket connection:

      Example Code:

      const socket = new WebSocket('ws://localhost:8080/ws');

socket.onmessage = function(event) {
    const message = JSON.parse(event.data);
    console.log(`${message.username}: ${message.content}`);
};

      Send a message to the server:

      socket.send(JSON.stringify({ username: "John", content: "Hello!" }));

  • Microservices (50 баллов)

    Микросервис: Управление транзакциями с генерацией фискального чека

    Микросервисы — это подход к архитектуре программного обеспечения, при котором приложение разделяется на небольшие, независимые сервисы. Каждый микросервис отвечает за определённую функциональность и взаимодействует с другими через API. Это позволяет разрабатывать, масштабировать и развертывать микросервисы независимо. Например, если Kaspi Bank реализует Kaspi Travel и Kaspi Магазин как отдельные микросервисы, то при сбое одного из них остальные продолжат работать без сбоев.

    Цель:

    Разработать микросервис для управления транзакциями при покупке товаров, услуг или премиум подписок с возможностью генерации фискального чека и отправки его на электронную почту клиента.

    Задачи на основном сервере:

    1. Создание страницы корзины (при необходимости):

      Добавьте необходимые страницы для оформления заказов и новые таблицы для корзины.

    1. Создание кнопки:

      Реализуйте функциональность оплаты услуг (например, оплатить корзину, если это магазин, или оформить премиум подписку, если логика магазина не подходит).

    1. Создание транзакции:

      После формирования корзины создайте отдельную таблицу для транзакций, которая будет содержать ID транзакции, ID корзины и их статус (например, "в ожидании оплаты", "оплачено", "отклонено"). Это позволит отслеживать статус заявки и понять, был ли счет оплачен.

    1. POST-запрос с основного сервера → в микросервис:

      После нажатия кнопки оплаты и создания ID транзакции отправьте POST-запрос из основного сервера в микросервис с данными корзины и клиента для создания записи в базе данных микросервиса. Пример запроса:

      {
  "cartItems": [
    {
      "id": "1",
      "name": "Product 1",
      "price": 100
    },
    {
      "id": "2",
      "name": "Product 2",
      "price": 200
    }
  ],
  "customer": {
    "id": "123",
    "name": "John Doe",
    "email": "john.doe@example.com"
  }
}

      Эти данные понадобятся в следующих заданиях.

    1. Получение ответа от микросервиса:

      В зависимости от ответа микросервиса ("true" или "false"), отображайте соответствующую информацию, например, на странице профиля пользователя.

    Задачи на микросервисе:

    1. Создание микросервиса:

      Разработайте отдельный микросервис на Go (или другом языке). Убедитесь, что он запускается отдельно от основного сервера, например, на порту 8081.

    1. Работа с базой данных:

      Микросервис должен на самом деле иметь собственную базу данных. Однако, для нашего небольшого проекта можно использовать ту же базу данных, но с разделением таблиц. Убедитесь, что нет зависимостей (foreign key) между таблицами основного сервера и микросервиса.

    1. Форма оплаты:

      После получения POST-запроса от основного сервера микросервис должен предоставить пользователю форму для ввода данных оплаты (номер карты, срок действия и т.д.). Для тестирования используйте фиктивные данные, возвращая успешный результат (true) или отказ в оплате (false) в зависимости от введённых данных (например, истекший срок действия карты).

      "paymentForm": {
    "cardNumber": "1234 5678 9012 3456",
    "expirationDate": "12/24",
    "cvv": "123",
    "name": "John Doe",
    "address": "123 Main Street"
}

      Результат оплаты необходимо вернуть основному серверу, который обновляет статус в таблице транзакций:

      • "в ожидании оплаты" → "оплачено" (если true),
      • "в ожидании оплаты" → "отклонено" (если false).
    1. Генерация фискального чека:

      После успешной оплаты сгенерируйте фискальный чек в формате PDF. Чек должен содержать следующую информацию:

      • Название компании или проекта,
      • Номер транзакции,
      • Дата и время оформления заказа,
      • Наименование товаров/услуг,
      • Цена за единицу,
      • Количество,
      • Общая сумма,
      • ФИО клиента,
      • Способ оплаты (с зашифрованными данными карты).
    1. Отправка чека:

      Отправьте сгенерированный PDF-чек на электронную почту клиента через микросервис.

    1. Завершение транзакции:

      Обновите статус транзакции как завершённый после успешной оплаты. Убедитесь, что изменения корректно отражены как в базе данных, так и в интерфейсе пользователя.

  • Websocket (50 баллов)

    Цель:

    Расширить функциональность вашего текущего проекта, реализовать чат, в котором клиенты могут создавать чат с поддержкой и общаться с администратором в режиме реального времени. Этот чат может быть полезен для быстрого взаимодействия клиентов с поддержкой, например, для решения вопросов по заказам или подпискам.

    Шаги:

    1. Использование текущего проекта:

      Проект уже включает административную панель и клиентскую страницу, их нужно дополнить функциями для реализации чата.

    1. Основные функции чата:

      Для реализации чата необходимо выполнить следующие шаги:

      1. Присвоение уникального идентификатора:

        Когда клиент открывает чат, ему автоматически присваивается уникальный идентификатор чата.

      1. Ограничение на один активный чат:

        Клиент может иметь только один активный чат одновременно.

      1. Отображение ссылок на чат:
        • Клиент видит ссылку на свой чат в личном профиле.
        • Администратор видит ссылки на чаты от разных клиентов в своей панели.
      1. Сохранение истории переписки:

        Чат должен сохранять историю сообщений, чтобы она была доступна после повторной авторизации, обновления страницы или завершения сессии. История должна сохраняться в базе данных или текстовом файле и быть привязана к уникальному идентификатору чата.

      1. Закрытие чата администратором:
        • Администратор может закрыть чат с клиентом.
        • После закрытия:
          • История переписки становится недоступной.
          • Устанавливается статус чата: «Активный» или «Неактивный».
          • У администратора исчезает ссылка на закрытый чат.
          • У клиента исчезает ссылка на чат, и снова появляется кнопка для создания нового чата.
      1. Создание нового чата:
        • После закрытия старого чата клиент может создать новый.
        • Для нового чата генерируется уникальный идентификатор, создается новая запись в базе данных, и процесс повторяется.
    1. Обновления в проекте:

      Для реализации чата потребуется два изменения в текущем проекте и одна новая страница:

      1. Страница чата:
        • В чате должны быть стандартные функции: отображение отправителя сообщения, времени отправки, содержания сообщения.
        • Обязательна кнопка отправки сообщения.
        • Визуально и функционально чат должен быть классическим и понятным.
      1. Изменения в профиле клиента:
        • Если у клиента есть активный чат, показывать ссылку на него.
        • Если активного чата нет, отображать кнопку для создания нового чата.
        • Не показывать ссылки на неактивные чаты.
      1. Изменения в админке:
        • Добавить вкладку или таблицу с ссылками на все активные чаты клиентов.
        • Добавить кнопку для закрытия чатов рядом с каждой ссылкой.

    Дополнительно:

    • Для работы с веб-сокетами используйте библиотеку github.com/gorilla/websocket.
    • На сервере: создайте обработчик /ws, принимайте и отправляйте сообщения через gorilla/websocket.

      Пример кода:

    CheckOrigin: func(r *http.Request) bool { return true },
}

        return
    }
    defer ws.Close()
}

      Убедитесь, что сервер слушает определённый порт:

      http.HandleFunc("/ws", handleConnections)
log.Fatal(http.ListenAndServe(":8080", nil))
    • На клиенте: используйте WebSocket API для подключения к серверу и отправки/приёма сообщений. В HTML-файле создайте подключение к веб-сокету:

      Пример кода:

socket.onmessage = function(event) {
