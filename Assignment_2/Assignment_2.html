    • 4.2 Error Handling (8 points)

      Error Handling in programming is the process of managing and responding to situations where a program encounters unexpected issues. In the context of Go, it’s not just about preventing errors but rather the ability to detect, interpret, and handle them correctly. This ensures that the program can continue running, inform the user about the error, or terminate gracefully.

      1. Error checking after each action:
        In Go, the standard approach involves functions returning an error as one of their results. The correct approach is to check for an error after each action.
        func doSomething() error {
    _, err := someOperation()
    if err != nil {
        // Error handling
        return fmt.Errorf("someOperation failed: %v", err)
    }
    // Further actions if there was no error
    return nil
}
      1. Error handling at a higher level:
        Instead of handling an error right where it occurs, it's sometimes better to pass it up to a higher level where it can be managed centrally.
        func doSomething() error {
    // Example function returning an error
    return errors.New("an error occurred")
}

func main() {
    err := doSomething()
    if err != nil {
        // Logging the error occurred in doSomething
        log.WithField("error", err).Error("Error occurred while executing doSomething")
    }
}

    • 4.3 Rate Limiting (9 points)
      1. Using NewLimiter for Rate Limiting:

        Implement rate limiting for your project's API to prevent overload. Use the golang.org/x/time/rate package for implementing the limiter.

        import (
    "golang.org/x/time/rate"
    "net/http"
)

var limiter = rate.NewLimiter(1, 3) // Rate limit of 1 request per second with a burst of 3 requests

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        if !limiter.Allow() {
            // Exceeded request limit
            http.Error(w, "Rate limit exceeded", http.StatusTooManyRequests)
            return
        }
        // Request processing
    })
    http.ListenAndServe(":8080", nil)
}
      1. Actions When Exceeding the Limit:

        When the number of requests exceeds the set limit, the server usually takes the following actions:

        • Returning an Error Code: The most common approach is to return an error to the client with the HTTP code 429 (Too Many Requests). This informs the client that the request limit has been exceeded.
        • Delaying Request Processing: In some cases, the server may not reject excess requests but simply delay their processing until the next permissible time interval for processing.
        • Providing Limit Information: It's good practice to send information in the response headers about how many more requests can be made before reaching the limit, and when the limit will be reset. (Suitable for those involved in file processing)
        • Offering Paid Subscription Plans: For commercial services, rate limiting is often associated with different subscription levels, where higher plans offer a larger number of permissible requests. (Also, for file processing)
        • Request Queuing: In some systems, requests exceeding the limit can be queued and processed later when possible.

        The goal of Rate Limiting is not just to block excessive traffic, but to manage it in such a way that both the server and clients operate efficiently and stably. Now, you just need to choose the most suitable option for your needs :)

      (This step is optional) Theoretical introduction.

      1. Preparing to Receive Shutdown Signals:

        Use the os/signal package to handle signals, such as SIGINT (Ctrl+C) and SIGTERM (a signal from the system to terminate the process). signal.Notify(quit, os.Interrupt, syscall.SIGTERM) instructs Go to listen for interrupt and termination signals.

        quit := make(chan os.Signal, 1)
signal.Notify(quit, os.Interrupt, syscall.SIGTERM)
      1. Launching the Web Server in a Separate Goroutine:

        1. Start your HTTP server in a separate goroutine so that the main goroutine can continue listening for signals.

        srv := &http.Server{Addr: ":8080"}

go func() {
    if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        log.Fatalf("ListenAndServe(): %v", err)
    }
}()
      1. Handling the Shutdown Signal:

        1. Upon receiving the shutdown signal, initiate the Graceful Shutdown process. The srv.Shutdown(ctx) method gracefully stops the server without interrupting ongoing request handling. context.WithTimeout ensures that the server will wait for the completion of all processes within the specified time (in this case, 30 seconds) before a complete shutdown.

        <-quit
log.Println("Server is shutting down...")

ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()

if err := srv.Shutdown(ctx); err != nil {
    log.Fatalf("Server forced to shutdown: %v", err)
}

log.Println("Server exiting")

        Graceful Shutdown is important for preventing data loss and ensuring proper termination of operations. It guarantees that all active connections and processes will be correctly completed before the application fully stops.

    • Graceful Shutdown (Корректное завершение работы): Это процесс безопасного останова приложения, при котором завершаются все текущие операции, сохраняются необходимые данные и освобождаются ресурсы перед полной остановкой.

    • 4.2 Error Handling (8 баллов)

      Error Handling (обработка ошибок) в программировании — это процесс управления и реагирования на ситуации, когда программа сталкивается с непредвиденными проблемами. В контексте Go это не просто предотвращение ошибок, а скорее умение обнаружить, интерпретировать и корректно обработать их, чтобы программа могла продолжить работу, сообщить пользователю об ошибке или завершиться корректным образом.

      1. Проверка ошибок после каждого действия:
        В Go, стандартный подход заключается в том, что функции возвращают ошибку в качестве одного из своих результатов. Правильный подход заключается в проверке наличия ошибки после каждого действия.
        func doSomething() error {
    _, err := someOperation()
    if err != nil {
        // Error handling
        return fmt.Errorf("someOperation failed: %v", err)
    }
    // Further actions if there was no error
    return nil
}
      1. Обработка ошибок на высшем уровне:
        Вместо того чтобы обрабатывать ошибку непосредственно там, где она возникла, иногда лучше передать её на верхний уровень, где она может быть обработана централизованно.
        func doSomething() error {
    // Example function returning an error
    return errors.New("an error occurred")
}

func main() {
    err := doSomething()
    if err != nil {
        // Logging the error occurred in doSomething
        log.WithField("error", err).Error("Error occurred while executing doSomething")
    }
}

      Для лучшего понимания приведу пример. Предположим, у нас уже реализованы фильтрация, сортировка и пагинация. В этом случае необходимо добавить обработку ошибок при фильтрации и сортировке. Если результат фильтрации или сортировки оказывается пустым, следует возвращать ошибку с сообщением: "No products match the filter" и отображать её клиенту, вместо того чтобы возвращать NULL или полагаться на стандартные сообщения от фронтенда.

    • 4.3 Rate Limiting (9 баллов)
      1. Использование NewLimiter для ограничения скорости:

        Реализуйте ограничение частоты запросов для API вашего проекта, чтобы предотвратить его перегрузку. Используйте пакет golang.org/x/time/rate для реализации ограничителя.

        import (
    "golang.org/x/time/rate"
    "net/http"
)

var limiter = rate.NewLimiter(1, 3) // Rate limit of 1 request per second with a burst of 3 requests

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        if !limiter.Allow() {
            // Exceeded request limit
            http.Error(w, "Rate limit exceeded", http.StatusTooManyRequests)
            return
        }
        // Request processing
    })
    http.ListenAndServe(":8080", nil)
}
      1. Действия при превышении лимита:

        Когда количество запросов превышает установленный лимит, сервер обычно предпринимает следующие действия:

        • Возвращение кода ошибки: Самый распространенный подход — возвращать клиенту ошибку с HTTP-кодом 429 (Too Many Requests). Это информирует клиента о том, что лимит запросов был превышен.
        • Задержка обработки запросов: В некоторых случаях сервер может не отклонять лишние запросы, а просто задерживать их обработку до тех пор, пока не наступит следующий разрешенный интервал времени для обработки.
        • Предоставление информации о лимите: Хорошая практика — отправлять в заголовках ответа информацию о том, сколько запросов можно еще сделать до достижения лимита, а также когда лимит будет сброшен. (Подойдет для тех, кто занимается обработкой файлов)
        • Предоставление платных тарифных планов: Для коммерческих сервисов ограничение скорости часто связано с разными уровнями подписки, где более высокие тарифные планы предоставляют большее количество допустимых запросов. (Так же, для обработки файлов)
        • Очереди запросов: В некоторых системах запросы, превышающие лимит, могут быть поставлены в очередь и обработаны позже, когда это станет возможным.

        Цель Rate Limiting — не просто блокировать избыточный трафик, а управлять им таким образом, чтобы и сервер, и клиенты функционировали эффективно и стабильно. Ну а вам, остается лишь выбрать подходящий вариант :)

    • 4.4 Graceful Shutdown

      (Данный пункт не обязательный) Теоретические ознакомление.

      1. Подготовка к Приему Сигналов Завершения:

        Используйте пакет os/signal для приема сигналов, например SIGINT (Ctrl+C) и SIGTERM (сигнал от системы для завершения процесса). signal.Notify(quit, os.Interrupt, syscall.SIGTERM) говорит Go слушать сигналы прерывания и завершения работы.

        quit := make(chan os.Signal, 1)
signal.Notify(quit, os.Interrupt, syscall.SIGTERM)
      1. Запуск Веб-сервера в Отдельной Горутине:

        Запустите ваш HTTP-сервер в отдельной горутине, чтобы основная горутина могла продолжать слушать сигналы.

        srv := &http.Server{Addr: ":8080"}

go func() {
    if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        log.Fatalf("ListenAndServe(): %v", err)
    }
}()
      1. Обработка Сигнала Завершения:

        При получении сигнала завершения начните процесс Graceful Shutdown. Метод srv.Shutdown(ctx) плавно останавливает сервер, не прерывая текущие обработки запросов. context.WithTimeout гарантирует, что сервер будет ожидать завершения всех процессов в течение указанного времени (в данном случае 30 секунд) перед полным выключением.

        <-quit
log.Println("Server is shutting down...")

ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()

if err := srv.Shutdown(ctx); err != nil {
    log.Fatalf("Server forced to shutdown: %v", err)
}

log.Println("Server exiting")

        Graceful Shutdown важен для предотвращения потери данных и корректного завершения работы. Он гарантирует, что все активные соединения и процессы будут корректно завершены, прежде чем приложение полностью остановится.
